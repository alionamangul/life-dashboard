## next.config.ts — критично для загрузки файлов
В Next 16 ДВА лимита тела запроса, оба подняты до 100 МБ (иначе фото/сканы с телефона не грузятся):
```ts
experimental: {
  serverActions: { bodySizeLimit: "100mb" }, // по умолч. 1 МБ
  proxyClientMaxBodySize: 100 * 1024 * 1024, // по умолч. 10 МБ (тело через proxy.ts)
}
```

## Грабли и фиксы (Next 16 / Prisma 6) — НЕ наступать снова
- **Загрузка файлов:** нужны ОБА лимита (`serverActions.bodySizeLimit` + `proxyClientMaxBodySize`), см. next.config.
- **Prisma 6:** `where:{ field:{ not:null } }` в `findMany` НЕ работает (фильтровать null в JS); в `updateMany`/`NOT:{field:null}` работает. Явный `field:null` в `create` отвергается, если поле required (был баг: `Document.category` ошибочно required → сделан `String?`).
- **Миграции:** `prisma migrate dev` блокируется в неинтерактивной среде на ДЕСТРУКТИВНЫХ изменениях (drop/rename column) → для таких писать `migration.sql` руками + `prisma migrate deploy`. Аддитивные (новые таблицы/колонки) — `migrate dev` проходит. После генерации клиента **перезапускать dev-сервер** (держит старый клиент в памяти).
- **Hydration mismatch:** клиентский фильтр из URL читать в `useEffect`, НЕ в `useState`-инициализаторе.
- **Оптимистичные галочки:** использовать функциональный `setState(prev => …)` updater — иначе при быстрых кликах подряд засчитывается только последний.
- **Скриншоты превью** ломались при схлопнутом вьюпорте → `preview_resize` mobile.
